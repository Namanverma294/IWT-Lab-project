Product Requirements Document (PRD)
Version: 1.0
Date: April 17, 2026
Status: Draft
Classification: Internal
Prepared for: Event Management Team
1. Executive Summary
The Event Management System (EMS) is a multi-role, web-based platform designed to streamline the creation, management, and participation of events within an organization. The platform serves three distinct user layers — public visitors, event participants, and internal organizing teams — each with precise access controls and role-specific dashboards.
The system draws its data model from a robust ER diagram encompassing entities such as Events, Customers, Organizers, Budget, Services (Photography, Venue, Catering, Music, Decoration), and Special Offers. It supports complex workflows from event discovery and participant registration to budget management, service allocation, and post-event feedback collection.
2. Project Objectives
Provide a publicly accessible event discovery interface requiring no login.
Enable seamless participant onboarding via signup, login, and personalized dashboards.
Deliver role-based access for event organizers, OC members, faculty advisors, and 4th-year student coordinators.
Maintain a three-layer database architecture reflecting the complexity of event, budget, service, and user relationships.
Fix the known signup-to-login redirection bug.
Support post-event feedback loops between participants and organizers.
Integrate service management (Photography, Venue, Catering, Music, Decoration) tied to events.
Enable budget tracking with special offers, payment records, and financial summaries.

9.2 Architecture Overview
The system follows a classic 3-tier architecture:
Presentation Layer: HTML + CSS + JavaScript files served as static resources. The JS fetch() API sends HTTP requests with JSON bodies to the backend.
Business Logic Layer: Jakarta Servlets receive HTTP requests, parse JSON using Gson, apply role-based logic, call the DAO layer, and write JSON responses using response.setContentType('application/json').
Data Layer: JDBC DAOs (Data Access Objects) execute SQL via PreparedStatement against MySQL. Connection pooling is handled by Apache DBCP2 (configured in context.xml).
ℹ No ORM (Hibernate/JPA) is used. All SQL is hand-written in DAO classes using JDBC PreparedStatement to prevent SQL injection.

9A.8 JDBC DAO Pattern
Each Servlet delegates DB operations to a DAO class. The pattern is:
Servlet reads JSON from request.getReader() using Gson.fromJson().
Servlet validates input fields; returns 400 with error JSON if invalid.
Servlet calls the appropriate DAO method (e.g., EventDAO.createEvent(event)).
DAO gets a Connection from DBUtil.getConnection() (Apache DBCP2 pool).
DAO executes a PreparedStatement with bound parameters.
DAO returns a model object or boolean to the Servlet.
Servlet serializes the result to JSON using Gson.toJson() and writes to response.getWriter().
Connection is closed in a finally block to return it to the pool.
⚠ All SQL uses PreparedStatement with ? placeholders. String concatenation in SQL is strictly forbidden to prevent SQL injection.
9A.9 Session-Based Authentication Flow (Jakarta HttpSession)
User submits login JSON to POST /api/auth/login.
LoginServlet queries DB via CustomerDAO / OrganizerDAO using JDBC.
Password is hashed with SHA-256 and compared to stored hash.
On match, servlet calls request.getSession(true) to create a session.
Session attributes stored: session.setAttribute('userId', id), session.setAttribute('role', role).
All protected endpoints read session.getAttribute('role') for RBAC.
AuthFilter.java intercepts all /api/* requests; redirects to /login if session is null.
Logout calls session.invalidate().
ℹ Session timeout is configured in web.xml: <session-config><session-timeout>30</session-timeout></session-config> (30 minutes).
9A.10 Frontend JavaScript Fetch Pattern
All frontend API calls use the native fetch() API with JSON. A shared api.js utility handles the base URL, headers, and error parsing:
Operation
JS fetch() Pattern
POST (Create/Login)
fetch('/api/auth/login', { method:'POST', headers:{'Content-Type':'application/json'}, body: JSON.stringify({ email, password }), credentials:'include' })
GET (List)
fetch('/api/events?month=4&year=2026', { credentials:'include' }).then(r => r.json())
PUT (Update)
fetch('/api/events/'+id, { method:'PUT', headers:{'Content-Type':'application/json'}, body: JSON.stringify(updates), credentials:'include' })
DELETE
fetch('/api/events/'+id, { method:'DELETE', credentials:'include' })
Error Handling
Check response.ok; if false, parse response.json() to get { success:false, error:'...' } and display to user
ℹ credentials:'include' is required on all fetch() calls so the browser sends the JSESSIONID cookie with each request to maintain the Jakarta HttpSession.
10. Non-Functional Requirements
10.1 Performance
Home page must load in under 2 seconds on a 4G connection.
All dashboard API responses must complete within 500ms for p95 requests.
Database queries must use indexes on F_start_date, C_email, O_id.
10.2 Security
All passwords must be hashed with SHA-256 before storing. Never store plaintext passwords.
All SQL must use PreparedStatement with bound parameters — no string concatenation in SQL queries.
Jakarta HttpSession is used for auth state. Sessions must be invalidated on logout via session.invalidate().
AuthFilter.java must intercept all /api/* routes and verify session attributes before passing the request to any Servlet.
JSON inputs must be validated in the Servlet layer before reaching the DAO layer; reject unexpected fields.
Response headers must include: X-Content-Type-Options: nosniff, X-Frame-Options: DENY to prevent XSS and clickjacking.
HTTPS enforced in production via Tomcat SSL connector; HTTP Strict Transport Security (HSTS) headers set.
CORS headers restricted to the frontend origin only, configured in a CORS Filter registered in web.xml.
10.3 Scalability
Database schema must support pagination for Events and Participant tables.
Stateless backend architecture to support horizontal scaling.
10.4 Accessibility
WCAG 2.1 AA compliance for all public-facing pages.
All form fields must have accessible labels and ARIA attributes.
